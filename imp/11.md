#### Impacto y riesgos

Se deja listo el siguiente bloque de la secuencia: **API mínima de citas para paciente y profesional**, sin tocar base de datos ni migraciones.

Este bloque agrega:

* listado público de profesionales aprobados;
* detalle público del profesional;
* cálculo de slots disponibles por fecha;
* creación de cita por paciente;
* listado y cancelación de citas del paciente;
* listado de citas del profesional.

Riesgos controlados:

* no se modifica el esquema;
* no se cambia el modelo `Appointment`;
* la reserva solo acepta horarios de 30 minutos;
* la reserva valida disponibilidad activa, bloqueos y citas activas previas;
* la cancelación solo aplica a citas activas del propio paciente.

Publicar los endpoints públicos y de reserva de citas.

**Riesgo controlado**

**Archivo a tocar**

`backend/apps/appointments/api_views.py`

`backend/apps/appointments/api_urls.py`
Acción: **reemplazo completo**

**Código exacto**

* [api_views.py](sandbox:/mnt/data/appointments_booking_api_patch/backend/apps/appointments/api_views.py)
* [api_urls.py](sandbox:/mnt/data/appointments_booking_api_patch/backend/apps/appointments/api_urls.py)

**Rutas nuevas**

* `GET /api/v1/professionals/public`
* `GET /api/v1/professionals/public/<professional_id>`
* `GET /api/v1/professionals/public/<professional_id>/available-slots?date=YYYY-MM-DD`
* `GET /api/v1/patient/appointments`
* `POST /api/v1/patient/appointments`
* `POST /api/v1/patient/appointments/<appointment_id>/cancel`
* `GET /api/v1/professional/appointments`

**Cómo probar**

Listar profesionales:

```bash
curl http://127.0.0.1:8000/api/v1/professionals/public
```

Ver slots:

```bash
curl "http://127.0.0.1:8000/api/v1/professionals/public/1/available-slots?date=2026-04-18&modality=in_person"
```

Crear cita:

```bash
curl -X POST http://127.0.0.1:8000/api/v1/patient/appointments \
  -H "Authorization: Bearer TOKEN_PACIENTE" \
  -H "Content-Type: application/json" \
  -d "{\"professional_id\":1,\"scheduled_at\":\"2026-04-18T08:00:00-05:00\",\"modality\":\"in_person\",\"notes\":\"Primera consulta\"}"
```

**Qué debe seguir funcionando**

* login;
* registro de paciente;
* registro de profesional;
* CRUD de disponibilidad del profesional.

**Cómo revertir esta fase**

Restaurar los archivos previos:

* `backend/apps/appointments/api_views.py`
* `backend/apps/appointments/api_urls.py`

#### Prueba de Fase 1

Se espera que:

* solo salgan profesionales `approved`, públicos y aceptando pacientes;
* los slots no muestren horarios ya tomados;
* los slots no muestren horarios bloqueados;
* una cita cancelada por paciente deje libre ese horario para futura reserva.
