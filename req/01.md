#### Impacto y riesgos

El objetivo es dejar una base inicial fuerte, rápida de desarrollar y fácil de mantener: **Django fuera de Docker para desarrollo ágil** y **PostgreSQL dentro de Docker con volumen persistente**. Django permite recarga automática del código durante desarrollo con `runserver`, por lo que los cambios se reflejan sin reinicios manuales en la mayoría de casos. Docker recomienda usar **volúmenes nombrados** para persistir los datos más allá del ciclo de vida del contenedor. Compose también permite controlar mejor el arranque con `healthcheck` y `depends_on`, lo que ayuda a evitar carreras al iniciar servicios. ([Django Project][1])

Riesgos controlados:

* Si se mete también Django en Docker desde el inicio, el desarrollo diario suele volverse más lento por rebuilds, sincronización de archivos y debugging más pesado.
* Si no se usa volumen para PostgreSQL, los datos pueden perderse al recrear el contenedor.
* Si no se separa configuración de desarrollo y producción, después se termina mezclando `runserver` con despliegue real.
* Si no se define una estructura mínima desde el primer día, el programador junior empieza a improvisar carpetas, settings y dependencias.

Supuestos conservadores:

* El proyecto inicia desde cero o casi cero.
* La prioridad actual es **velocidad de desarrollo**, no despliegue final.
* No se va a construir todavía microservicios ni arquitectura distribuida.
* La base inicial debe servir para crecer después hacia Redis, Celery, Nginx y producción seria, sin rehacer todo.

#### Preparación

Antes de tocar nada, se sugiere dejar estas decisiones cerradas y no negociables:

1. **Framework base:** Django.
2. **Base de datos:** PostgreSQL.
3. **Modo desarrollo:** Django local con `runserver`.
4. **Modo datos:** PostgreSQL en Docker con volumen persistente.
5. **Configuración:** variables de entorno en archivo `.env`.
6. **Separación de ambientes:** `development` y `production`.
7. **Primera meta:** levantar proyecto, conectar base, correr migraciones, crear superusuario y probar admin.

Backups y protección:

* Si ya existe algo, crear rama antes de empezar:

```bash
git checkout -b base-inicial-django
```

* Si ya existe una base previa en Docker, no ejecutar `docker compose down -v`.
* Si ya existe código, copiar respaldo del estado inicial:

```bash
git add .
git commit -m "backup antes de base inicial django"
```

#### Fase 1

**Objetivo**
Levantar PostgreSQL en Docker con persistencia real.

**Riesgo controlado**
Evitar pérdida de datos al reiniciar o recrear el contenedor.

**Archivo a tocar**
`infra/docker-compose.yml`

**Acción**
Crear este archivo exacto:

```yaml
services:
  db:
    image: postgres:16
    container_name: proyecto_postgres
    restart: unless-stopped
    environment:
      POSTGRES_DB: appdb
      POSTGRES_USER: appuser
      POSTGRES_PASSWORD: apppass
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U appuser -d appdb"]
      interval: 10s
      timeout: 5s
      retries: 5

volumes:
  postgres_data:
```

**Cómo probar**
Ejecutar:

```bash
docker compose -f infra/docker-compose.yml up -d
docker ps
docker volume ls
```

Luego:

```bash
docker exec -it proyecto_postgres psql -U appuser -d appdb
```

**Qué debe seguir funcionando**
Si ya existía Docker en la máquina, no debe tocar otros proyectos. Solo debe levantarse este contenedor y su volumen.

**Cómo revertir esa fase**
Detener solo el stack:

```bash
docker compose -f infra/docker-compose.yml down
```

No usar `-v` si se quiere conservar la data.

---

#### Prueba de Fase 1

Resultado esperado:

* El contenedor `proyecto_postgres` debe quedar en estado `Up`.
* El volumen `postgres_data` debe existir.
* Debe ser posible entrar con `psql`.
* Si se reinicia el contenedor, la base debe seguir existiendo.

Prueba de persistencia:

```bash
docker compose -f infra/docker-compose.yml restart
docker exec -it proyecto_postgres psql -U appuser -d appdb
```

La persistencia con volúmenes nombrados está documentada por Docker como el mecanismo estándar para conservar datos fuera del contenedor. ([Docker Documentation][2])

#### Fase 4

**Objetivo**
Levantar Django local con autoreload y dejar listo el flujo mínimo de desarrollo.

**Riesgo controlado**
Asegurar que el programador vea cambios rápidos al guardar.

**Archivo a tocar**
`README.md`

**Contenido mínimo sugerido**

```md
## Desarrollo local

### 1. Levantar PostgreSQL
docker compose -f infra/docker-compose.yml up -d

### 2. Activar entorno virtual
source .venv/bin/activate

### 3. Entrar al backend
cd backend

### 4. Migrar
python manage.py migrate

### 5. Crear superusuario
python manage.py createsuperuser

### 6. Levantar servidor
python manage.py runserver
```

**Cómo probar**
Ejecutar:

```bash
cd backend
python manage.py runserver
```

Abrir:

* `http://127.0.0.1:8000/`
* `http://127.0.0.1:8000/admin/`

Cambiar una vista, guardar y verificar recarga.

Django documenta que el servidor de desarrollo recarga el código Python automáticamente en la mayoría de cambios, sin requerir reinicio manual, aunque algunos cambios como agregar archivos nuevos sí pueden requerir reinicio. ([Django Project][1])

**Qué debe seguir funcionando**
Debe seguir conectando a PostgreSQL y mostrando el admin.

**Cómo revertir esa fase**
Cerrar `runserver`. No afecta datos.

#### Pruebas de regresión

Como todavía es una base inicial, la regresión mínima debe validar esto:

1. La base no se pierde al reiniciar contenedor.
2. Django sigue levantando después de reiniciar la máquina.
3. El admin sigue autenticando.
4. Las migraciones nuevas se aplican sin romper las anteriores.
5. El proyecto no depende de Docker para correr Django en desarrollo.
6. Nadie está usando todavía configuración de producción por error.

También conviene ejecutar:

```bash
cd backend
python manage.py check
python manage.py test
```

Y revisar que `docker ps` muestre solo el PostgreSQL del proyecto.

La documentación oficial de Django diferencia claramente desarrollo y despliegue, y recomienda checklist específico para producción en una etapa posterior. ([Django Project][3])

#### Plan de reversión total

Si algo sale mal, la reversión total sería:

1. Detener Django local.
2. Bajar PostgreSQL sin borrar volumen:

```bash
docker compose -f infra/docker-compose.yml down
```

3. Si se quiere volver al estado previo del repo:

```bash
git reset --hard HEAD
```

4. Si el problema fue solo de código, mantener el volumen y rehacer el backend.
5. Si se quiere destruir totalmente la base de desarrollo:

```bash
docker compose -f infra/docker-compose.yml down -v
```

Advertencia: ese último comando sí borra la persistencia.

### Instrucción final masticada para el programador

El programador no debe improvisar arquitectura. Debe hacer exactamente esto, en este orden:

1. Crear `infra/docker-compose.yml` con PostgreSQL y volumen persistente.
2. Levantar PostgreSQL y comprobar acceso.
3. Crear `.venv` e instalar Django, psycopg y dotenv.
4. Crear proyecto `backend/config`.
5. Separar settings en `base.py`, `dev.py` y `prod.py`.
6. Conectar Django a PostgreSQL usando `.env`.
7. Correr migraciones.
8. Crear superusuario.
9. Levantar `runserver`.
10. Verificar admin y autoreload.
11. No meter todavía Redis, Celery, Nginx ni Docker para Django.

Con eso queda una base fuerte, rápida y conservadora para arrancar. La siguiente fase natural sería dejar **apps de negocio** como `users`, `professionals`, `appointments` y `billing`, pero sin tocar todavía la base inicial.
