#### Impacto y riesgos

Para crear las tablas de un proyecto Django sobre PostgreSQL, **la vía correcta no es escribir todo a mano en SQL desde el inicio**, sino:

1. definir los modelos en Django,
2. generar migraciones con `makemigrations`,
3. aplicar esas migraciones con `migrate`.

Django documenta que `makemigrations` crea los archivos de migración a partir de los modelos, `migrate` sincroniza la base con esas migraciones y `sqlmigrate` permite ver el SQL exacto que Django va a ejecutar. Además, `migrate` también crea las tablas necesarias del sistema de autenticación y permisos de Django para las apps instaladas. ([Django Project][1])

El riesgo más importante está en el usuario del sistema: **si se piensa usar login por email o un usuario personalizado, eso debe decidirse antes de la primera migración**, porque Django advierte que cambiar `AUTH_USER_MODEL` después de haber creado tablas es complejo y no se resuelve automáticamente. ([Django Project][2])

Supuesto conservador: como todavía no está cerrado el 100% del catálogo funcional, se propone **solo la base mínima necesaria para arrancar bien**: usuarios, pacientes, profesionales, especialidades, documentos del profesional, disponibilidad, citas, pagos y auditoría básica. No se incluyen todavía módulos más avanzados como laboratorio, moderación o privacidad completa, para no sobrediseñar.

from django.contrib.auth.base_user import BaseUserManager


class UserManager(BaseUserManager):
    use_in_migrations = True

    def _create_user(self, email, password, **extra_fields):
        if not email:
            raise ValueError("El email es obligatorio")
        email = self.normalize_email(email)
        user = self.model(email=email, **extra_fields)
        user.set_password(password)
        user.save(using=self._db)
        return user

    def create_user(self, email, password=None, **extra_fields):
        extra_fields.setdefault("is_staff", False)
        extra_fields.setdefault("is_superuser", False)
        extra_fields.setdefault("is_active", True)
        return self._create_user(email, password, **extra_fields)

    def create_superuser(self, email, password=None, **extra_fields):
        extra_fields.setdefault("is_staff", True)
        extra_fields.setdefault("is_superuser", True)
        extra_fields.setdefault("is_active", True)

        if extra_fields.get("is_staff") is not True:
            raise ValueError("El superusuario debe tener is_staff=True")
        if extra_fields.get("is_superuser") is not True:
            raise ValueError("El superusuario debe tener is_superuser=True")

        return self._create_user(email, password, **extra_fields)
```

from django.contrib.auth.models import AbstractBaseUser, PermissionsMixin
from .managers import UserManager


class User(AbstractBaseUser, PermissionsMixin):

        SUPER_ADMIN = "super_admin", "Super Admin"
        ADMIN = "admin", "Admin"
        PRIVACY_AUDITOR = "privacy_auditor", "Privacy Auditor"
        PROFESSIONAL = "professional", "Professional"

    email = models.EmailField(unique=True)
    first_name = models.CharField(max_length=120)
    last_name = models.CharField(max_length=120)
    phone = models.CharField(max_length=30, blank=True)
    role = models.CharField(max_length=30, choices=Role.choices)
    is_active = models.BooleanField(default=True)
    is_staff = models.BooleanField(default=False)
    date_joined = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    objects = UserManager()

    USERNAME_FIELD = "email"
    REQUIRED_FIELDS = ["first_name", "last_name", "role"]

    class Meta:
        db_table = "users_user"
        indexes = [

            models.Index(fields=["role"], name="idx_users_role"),

### Cómo se haría correctamente en PostgreSQL

La respuesta corta operativa sería esta:

**No se debería escribir manualmente todo el esquema en `CREATE TABLE` puro salvo casos excepcionales.**
Lo correcto sería:

1. definir modelos,
2. correr `makemigrations`,
3. revisar SQL con `sqlmigrate`,
4. aplicar con `migrate`.

Ese flujo existe justamente para que Django respete dependencias entre tablas y llaves foráneas. La documentación oficial explica que las migraciones manejan esas dependencias entre apps para crear primero la tabla referenciada y luego la clave foránea que depende de ella. ([Django Project][1])

3. aplicar esas migraciones con `migrate`.

Django documenta que `makemigrations` crea los archivos de migración a partir de los modelos, `migrate` sincroniza la base con esas migraciones y `sqlmigrate` permite ver el SQL exacto que Django va a ejecutar. Además, `migrate` también crea las tablas necesarias del sistema de autenticación y permisos de Django para las apps instaladas. ([Django Project][1])

El riesgo más importante está en el usuario del sistema: **si se piensa usar login por email o un usuario personalizado, eso debe decidirse antes de la primera migración**, porque Django advierte que cambiar `AUTH_USER_MODEL` después de haber creado tablas es complejo y no se resuelve automáticamente. ([Django Project][2])

Supuesto conservador: como todavía no está cerrado el 100% del catálogo funcional, se propone **solo la base mínima necesaria para arrancar bien**: usuarios, pacientes, profesionales, especialidades, documentos del profesional, disponibilidad, citas, pagos y auditoría básica. No se incluyen todavía módulos más avanzados como laboratorio, moderación o privacidad completa, para no sobrediseñar.

from django.contrib.auth.base_user import BaseUserManager


class UserManager(BaseUserManager):
    use_in_migrations = True

    def _create_user(self, email, password, **extra_fields):
        if not email:
            raise ValueError("El email es obligatorio")
        email = self.normalize_email(email)
        user = self.model(email=email, **extra_fields)
        user.set_password(password)
        user.save(using=self._db)
        return user

    def create_user(self, email, password=None, **extra_fields):
        extra_fields.setdefault("is_staff", False)
        extra_fields.setdefault("is_superuser", False)
        extra_fields.setdefault("is_active", True)
        return self._create_user(email, password, **extra_fields)

    def create_superuser(self, email, password=None, **extra_fields):
        extra_fields.setdefault("is_staff", True)
        extra_fields.setdefault("is_superuser", True)
        extra_fields.setdefault("is_active", True)

        if extra_fields.get("is_staff") is not True:
            raise ValueError("El superusuario debe tener is_staff=True")
        if extra_fields.get("is_superuser") is not True:
            raise ValueError("El superusuario debe tener is_superuser=True")

        return self._create_user(email, password, **extra_fields)
```

La respuesta corta operativa sería esta:

#### Impacto y riesgos

Para la carga inicial de catálogos y datos de prueba, la opción más conservadora no es meter esos datos en una migración automática, sino crear un **comando de gestión propio de Django**. Django soporta comandos personalizados ejecutables con `manage.py`, y también soporta fixtures y migraciones de datos con `RunPython`; pero para datos demo o de arranque controlado conviene no forzar su ejecución automática en todos los ambientes. Eso deja el sembrado explícito, repetible e idempotente. ([Django Project][1])

Riesgos controlados:

* No se toca el esquema de la base; solo se insertan datos.
* No se mezclan datos demo dentro de migraciones estructurales.
